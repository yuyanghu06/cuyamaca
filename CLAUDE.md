# CLAUDE.md — Cuyamaca

## Project Overview

Cuyamaca is a Tauri v2 desktop application for natural language control of Arduino-based robotics and hardware projects. Users define their hardware, upload or generate Arduino sketches, and then control their boards through conversational prompts backed by a locally-hosted or externally-hosted LLM.

The app handles the full lifecycle: manifest-driven code generation, model-assisted sketch modification with visible diffs, flashing via arduino-cli, and an agentic runtime control loop that reads structured serial output and sensor visualizations as model context. The entire stack runs locally by default. External model APIs are supported as opt-in.

---

## Core Concepts

### Project

The unit of work in Cuyamaca. A project contains:

- **Manifest** — the hardware definition. Board type, serial port, and a list of all connected components with their pin assignments, connection types, and subtypes.
- **Sketch** — the current `.ino` file. Either uploaded by the user or generated by the code model from the manifest.
- **Tool Registry** — synthesized tool definitions derived from the sketch. Maps callable tool names to serial commands the board understands.
- **Version History** — previous sketch versions with diffs, so the user can see every change the model has made and roll back if needed.

```
cuyamaca-projects/
  my-robot/
    manifest.json
    sketch.ino
    tools.json
    history/
      sketch_v1.ino
      sketch_v2.ino
      ...
```

### Manifest

The manifest is the model's ground truth for what the hardware can do. It is authored by the user in the parts editor UI, not inferred from code. All code generation and tool synthesis is grounded in the manifest.

```json

  "board": "arduino:avr:uno",
  "serial_port": "/dev/cu.usbmodem14201",
  "baud_rate": 115200,
  "components": [
    {
      "id": "motor_left",
      "type": "dc_motor",
      "pins": { "pwm": 5, "dir_a": 2, "dir_b": 3 },
      "label": "Left Drive Motor"
    },
    {
      "id": "motor_right",
      "type": "dc_motor",
      "pins": { "pwm": 6, "dir_a": 7, "dir_b": 8 },
      "label": "Right Drive Motor"
    },
    {
      "id": "ultrasonic_front",
      "type": "ultrasonic",
      "pins": { "trig": 9, "echo": 10 },

      "id": "servo_head",
```

### Two Model Roles

Two independent model slots. Each is configured separately in Settings.

**Code Model** — active during project mode. Reads the manifest, generates and modifies sketches, synthesizes tool definitions, produces diffs. Latency-insensitive — the user reviews output before anything is flashed. External APIs are a natural fit here since the best coding models may not run locally.

**Runtime Model** — active during the agentic control loop after flashing. Reads structured sensor context and sensor visualization images, decides tool calls, reacts to serial feedback. Must be multimodal. Latency-sensitive — it is in a live control loop. Local Ollama is the default. External APIs are supported as an escape hatch.

Both models are configured independently. A user can use Claude for code generation and LLaVA locally for runtime control, or GPT-4o for both, or any other combination.

## Architecture

### External Dependencies

- **arduino-cli** — compiles and flashes `.ino` sketches to the board. Used only at flash time, not at runtime. Managed as a child process by the Rust backend.
- **Ollama (MIT)** — local LLM/multimodal inference runtime. Default for both model slots. Managed as a child process.
- **External model APIs (optional)** — OpenAI, Anthropic, Google, Mistral. Used when the user configures a non-Ollama model for either slot. All API calls are made by the Rust backend. API keys are stored in the OS keychain via Tauri's secure storage.

### The App (Tauri v2)

Two API surfaces, both managed by the Rust backend:

**Internal API (Tauri Commands)** — the TypeScript frontend calls these exclusively. Every user action flows through a Tauri command. The frontend never calls Ollama, external APIs, the serial port, or arduino-cli directly.

**Serial Interface** — the Rust backend holds the serial port connection after flashing. Two concurrent operations: writing tool call commands to the board, and reading structured output from the board. The reader feeds two consumers simultaneously — the UI serial monitor and the runtime model context.

### Request Flow — Project Mode (pre-flash)

```
User defines manifest in parts editor
        │
        ▼
User uploads sketch OR requests generation
        │
        ▼
Rust backend passes manifest + sketch/request to code model
        │
        ▼
Code model generates or modifies sketch
        │
        ▼
Rust backend returns full sketch + diff to frontend
        │
        ▼
UI displays code block with highlighted diff
        │
        ▼
User approves → Rust backend invokes arduino-cli to flash
        │
        ▼
Runtime window opens
```

### Request Flow — Runtime Mode (post-flash)

```
User sends natural language prompt
        │
        ▼
Rust backend assembles context object:
  - structured sensor state (text)
  - sensor visualization images (PNG)
  - camera frame (JPEG, if present)
  - available tool definitions
        │
        ▼
Runtime model receives context, decides tool call(s)
        │
        ▼
Rust backend writes serial command to board
        │
        ▼
Board executes, prints structured serial output
        │
        ▼
Rust backend reads serial output:
  → parses into sensor state
  → streams raw lines to serial monitor UI
  → feeds updated state back into model context
        │
        ▼
Model reads new state, decides next action or terminates loop
        │
        ▼
Repeat until user kills the loop
```

## Tech Stack

### Rust Backend (src-tauri/)
- Owns ALL orchestration logic
- Manages arduino-cli child process (compile + flash)
- Manages Ollama child process (local inference)
- Holds serial port connection (read + write concurrently)
- Parses structured serial output into sensor state
- Renders sensor visualization images (PNG) from sensor state
- Assembles multimodal context objects for runtime model
- Makes all external API calls (OpenAI, Anthropic, Google, Mistral)
- Reads and writes API keys via OS keychain (Tauri secure storage)
- Exposes Tauri commands for the frontend
- Handles platform-specific logic (macOS vs Windows)

### TypeScript Frontend (src/)
- Thin client — renders UI, calls Tauri commands, displays results
- Project editor (manifest parts editor, code view with diffs)
- Runtime window (chat interface, serial monitor, tool call pills)
- Settings (model configuration, API keys, port selection)
- Never calls external services, serial ports, or arduino-cli directly

## Serial Protocol

### Structured Output Convention

All serial output from the board follows a structured key-value format. When the code model generates or modifies a sketch, it must follow this convention for all `Serial.print` statements. The Rust backend parses lines matching this format and ignores malformed output.

```
SENSOR_ID:VALUE
SENSOR_ID:VALUE,VALUE,VALUE   (for multi-axis sensors)
```

Examples:
```
DIST_FRONT:14
ACCEL:0.12,0.03,9.81
GYRO:0.01,-0.02,0.00
HEADING:247.3
ENC_L:1842
ENC_R:1836
BUMP_L:0
BUMP_R:1
LINE:00011000
TEMP:22.4
LUX:340
STATE:moving_forward
BATT:7.4
```

The sensor ID must match the component `id` field from the manifest. The Rust backend uses the manifest to know how to interpret the value (units, range, type).

### Serial Command Format

Tool calls from the model are written to the board as:

```
CMD:tool_name:param1=value1,param2=value2\n
```

Examples:
```
CMD:move_forward:speed=80\n
CMD:turn_left:degrees=45,speed=60\n
CMD:set_servo:angle=90\n
CMD:stop\n
```

The sketch's `loop()` reads lines, parses the `CMD:` prefix, and dispatches to the appropriate function. The code model generates this dispatch logic when it writes or modifies the sketch.

## Sensor Types and Context Handling

### Scalar sensors → structured text

Single-value sensors are included in the model context as parsed text. The Rust backend aggregates the latest value for each sensor ID and formats a state block:

```
sensor state (sampled at 100ms intervals):
- front distance (DIST_FRONT): 14cm
- heading (HEADING): 247.3°
- left bumper (BUMP_L): clear
- right bumper (BUMP_R): triggered
- left encoder (ENC_L): 1842 ticks
- right encoder (ENC_R): 1836 ticks
- battery (BATT): 7.4v
- board state (STATE): moving_forward
```

### Spatial / array sensors → visualization image

Sensors with spatial structure are rendered as PNG images by the Rust backend and included in the model context as image inputs:

- **Line sensor array** (`LINE:00011000`) → bitmap showing active sensors, scaled to a readable size
- **Multi-ultrasonic sweep** → top-down arc view with distance markers
- **LIDAR point cloud** → 2D overhead map
- **IMU time-series** → line chart of recent accel/gyro values (last 2 seconds)
- **Encoder drift** → bar comparison of left vs right wheel distance

### Camera frames → direct image input

JPEG frames from camera components are passed directly to the runtime model as image inputs. The Rust backend receives frames over WiFi (ESP32-CAM) or serial and includes the latest frame in each context assembly.

### Context object assembled per model turn

```
[system prompt: role, available tools, manifest summary]

[text: structured sensor state block]

[image: sensor_visualization.png]  (if spatial sensors present)

[image: camera_frame.jpg]          (if camera component present)

[text: conversation history]

[text: user message]
```

## Tool Registry

### Structure

Tool definitions are synthesized by the code model when a sketch is generated or uploaded. Each tool maps a callable name to a serial command the board understands.

```json
{
  "tools": [
    {
      "name": "move_forward",
      "description": "Drive both motors forward. Use when the user wants the robot to move ahead. Speed is a percentage of maximum motor power.",
      "parameters": {
        "speed": {
          "type": "integer",
          "range": "0-100",
          "default": 60,
          "required": false
        }
      },
      "serial_command": "CMD:move_forward:speed={speed}"
    },
    {
      "name": "turn_left",
      "description": "Rotate the robot left in place by running motors in opposite directions. Degrees is approximate and depends on surface friction.",
      "parameters": {
        "degrees": {
          "type": "integer",
          "range": "0-360",
          "required": true
        },
        "speed": {
          "type": "integer",
          "range": "0-100",
          "default": 50,
          "required": false
        }
      },
      "serial_command": "CMD:turn_left:degrees={degrees},speed={speed}"
    },
    {
      "name": "stop",
      "description": "Immediately stop all motors. Use when the robot needs to halt, when an obstacle is detected, or when the user says stop.",
      "parameters": {},
      "serial_command": "CMD:stop"
    }
  ]
}
```

### Synthesis Rules

When the code model synthesizes tools from a sketch:

- Every serial-dispatchable function in the sketch gets a tool definition
- Tool names use `snake_case` and match the function name in the sketch
- Descriptions must be unambiguous enough for the runtime model to select the correct tool from natural language. Write them as if the runtime model has never seen the sketch.
- Parameters must match what the serial command actually accepts. Do not invent parameters the sketch does not handle.
- If the uploaded sketch has no serial dispatch loop, the code model must add one before synthesizing tools. The modified sketch must be shown to the user before flashing.

### Lifecycle Tools

In addition to domain tools, the runtime model has access to lifecycle tools managed by the Rust backend (not serial commands):

- `read_sensor_state` — returns the current parsed sensor state as text
- `wait_milliseconds` — pauses the agent loop for a specified duration
- `get_camera_frame` — requests a fresh camera frame (if camera component present)
- `end_session` — terminates the agentic loop and closes the runtime window

## Sketch Generation Rules

When the code model generates or modifies a sketch, it must follow these rules:

1. **Always include serial initialization** — `Serial.begin(115200)` in `setup()`. Baud rate must match the manifest.

2. **Always include a command dispatch loop** — `loop()` reads `Serial.readStringUntil('\n')`, parses the `CMD:` prefix, and dispatches to handler functions.

3. **Always use structured output** — every `Serial.print` must follow the `SENSOR_ID:VALUE` format. No freeform debug strings in production sketches.

4. **Print sensor state at fixed intervals** — use a `millis()`-based timer, not `delay()`. Default interval: 100ms. This keeps sensor output consistent without blocking the command loop.

5. **Include an emergency stop** — `CMD:stop` must always halt all actuators immediately. This is non-negotiable regardless of what else the sketch does.

6. **Pin assignments come from the manifest** — never hardcode pin numbers. Use `#define` or `const int` declarations at the top of the sketch that match the manifest exactly.

7. **Show all changes as diffs** — whenever the code model modifies an existing sketch (rather than generating from scratch), it must return both the full modified sketch and a structured diff. Nothing is flashed without user approval.

## Component Library

The parts editor exposes these component types. Each has a known pin configuration and known sensor output format. The code model uses this library to generate correct pin setup and sensor reading code.

### Actuators
- `dc_motor` — PWM + 2 direction pins. Tool: speed (-100 to 100, negative = reverse)
- `servo` — single signal pin. Tool: angle (0-180)
- `stepper_motor` — step + direction pins. Tool: steps, direction, speed
- `relay` — single digital pin. Tool: on/off
- `led` — single digital or PWM pin. Tool: on/off/brightness

### Distance / Proximity
- `ultrasonic` (HC-SR04) — trig + echo pins. Output: `SENSOR_ID:cm`
- `ir_distance` — analog pin. Output: `SENSOR_ID:0-1023`
- `lidar_serial` (TF-Mini) — RX/TX pins. Output: `SENSOR_ID:cm,strength`

### Motion / Orientation
- `imu` (MPU-6050) — I2C (SDA/SCL). Output: `ACCEL:x,y,z GYRO:x,y,z`
- `magnetometer` (HMC5883L) — I2C. Output: `HEADING:degrees`
- `encoder` — 2 digital pins. Output: `SENSOR_ID:ticks`

### Touch / Tactile
- `bump_switch` — single digital pin. Output: `SENSOR_ID:0` or `SENSOR_ID:1`
- `line_sensor_array` — N digital pins. Output: `SENSOR_ID:binary_string`
- `force_sensor` — analog pin. Output: `SENSOR_ID:0-1023`

### Environmental
- `temp_humidity` (DHT22) — single data pin. Output: `TEMP:c HUM:pct`
- `barometer` (BMP280) — I2C. Output: `PRES:hpa ALT:m`
- `light` (BH1750) — I2C. Output: `LUX:value`
- `gas` (MQ series) — analog pin. Output: `SENSOR_ID:ppm`

### Vision
- `camera` (ESP32-CAM) — WiFi connection, separate from Arduino serial. JPEG frames streamed to Rust backend over HTTP. Resolution and format specified in manifest.

## Build & Distribution

### Installer Format
- **macOS:** `.dmg`
- **Windows:** `.exe` or `.msi`

Tauri v2's bundler produces these. No terminal scripts.

### First-Run Setup

On first launch, the app detects whether Ollama and arduino-cli are installed. If either is missing, the app downloads and installs them automatically. The user never opens a terminal.

This logic is platform-specific with separate macOS and Windows code paths.

- **macOS:** Homebrew or direct binary downloads
- **Windows:** winget or direct installer downloads

### Process Lifecycle

- **On launch:** Start Ollama as a child process. Wait for it to be healthy before marking the app ready. arduino-cli is invoked on demand (not kept running).
- **On exit:** Gracefully shut down Ollama. Close any open serial connections. Do not leave orphaned processes.
- **On crash:** Handle unexpected Ollama termination. Surface error in the sidebar status indicator. Attempt restart or prompt user.

## UI Design

### Design Language

Cuyamaca uses a dark industrial-technical aesthetic — the visual language of electronics workbenches, oscilloscopes, and PCB design tools. It should feel like precision hardware, not consumer software. The palette is deep near-black backgrounds with electric accent colors: cyan for active states and sensor data, amber for warnings and motor activity, green for healthy connections, red for errors and stops.

Monospace type is used for all code, serial output, and sensor readings. It should feel native to the domain. Proportional type is used for prose UI (labels, descriptions, chat).

The background is not ambient or decorative — it is functional darkness that makes the data glow.

### Glass Hierarchy

Same three-tier system as Sierra, adapted for the dark industrial palette:

- **Subtle:** Structural chrome — sidebar, panel headers, status bar. `backdrop-filter: blur(12px); background: rgba(255,255,255,0.03); border: 0.5px solid rgba(255,255,255,0.06)`
- **Standard:** Interactive elements — component cards, input fields, chat bubbles. `backdrop-filter: blur(20px); background: rgba(255,255,255,0.06); border: 0.5px solid rgba(255,255,255,0.10)`
- **Strong:** Emphasis — active selections, modals, user messages. `backdrop-filter: blur(28px); background: rgba(255,255,255,0.10); border: 0.5px solid rgba(255,255,255,0.16)`

**Sidebar (left, 220px):** App identity at top. Project list below — each project as a named item with board type and connection status. Service health at bottom: Ollama, arduino-cli, active code model, active runtime model. Status dots: green = ready, amber = loading, red = unavailable.

### Code View

The code editor displays the full current sketch with syntax highlighting. When the code model produces a modification, the diff is overlaid using a side-by-side or inline diff format — added lines highlighted in cyan-tinted glass, removed lines in red-tinted glass. An "Approve & Flash" button appears at the top of the code view when a pending diff is present. A "Reject" button discards the modification and returns to the previous sketch.

The code view is read-only except through the model. Users do not edit the sketch directly — they ask the code model to make changes in the chat view.

### Runtime Window

The runtime window is a separate Tauri window that opens at flash time. It is distinct from the project window — closing it terminates the serial session but does not close the project.

```
┌─────────────────────────────────────┬──────────────────┐
│                                     │                  │
│  Chat (model + tool call pills)     │  Serial Monitor  │
│                                     │  (raw output)    │
│                                     │                  │
│                                     │ ──────────────── │
│                                     │                  │
│                                     │  Sensor State    │
│                                     │  (parsed, live)  │
│                                     │                  │
│                                     │ ──────────────── │
│                                     │                  │
│                                     │  Sensor Viz      │
│  ─────────────────────────────────  │  (images)        │
│  Input capsule          [KILL]      │                  │
└─────────────────────────────────────┴──────────────────┘
```

**Chat (left):** Natural language prompts and model responses. Tool call confirmation pills appear inline — cyan-tinted for serial writes, green-tinted for successful execution, red-tinted for errors. The model's agentic loop runs here — it may issue multiple tool calls per user prompt before settling on a final response.

**Serial Monitor (right top):** Raw serial output from the board, scrolling. Monospace font, 11px. Lines that match the structured output format are dimmed slightly — they're already parsed into the sensor state panel below. Unrecognized lines appear at full brightness (useful for debugging).

**Sensor State (right middle):** Live parsed sensor values, updated as new serial lines arrive. Each component from the manifest gets a row. Values animate on change (brief cyan flash on the value text).

**Sensor Visualizations (right bottom):** PNG images rendered by the Rust backend from spatial sensor data. Refreshes at the same interval as sensor state (100ms). If no spatial sensors or camera are present, this panel is hidden.

**Kill Button:** Large, always-visible red button. Terminates the agentic loop, sends `CMD:stop` to the board, closes the serial connection, and closes the runtime window. Keyboard shortcut: Escape.

## Key Design Principles

1. **Manifest is ground truth.** The code model generates from the manifest, not from assumptions. If something is not in the manifest, the model does not invent it.

2. **Nothing flashes without user approval.** Every sketch modification — whether generated from scratch or modified from an upload — is shown as a full code view with highlighted diff before the flash button becomes available.

3. **Structured serial output is enforced at generation time.** The code model writes the sketch to follow the `SENSOR_ID:VALUE` protocol. The Rust backend parser is strict. This is what makes sensor data usable as model context.

4. **Runtime model is multimodal only.** Text-only models cannot participate in the runtime loop. The app enforces this in the settings UI and model selector.

5. **Two model roles, independently configured.** Code model and runtime model are separate concerns. A user can mix local and external models freely across the two slots.

6. **Frontend is a thin client.** No external calls from TypeScript. No serial port access from TypeScript. No arduino-cli invocation from TypeScript. Everything goes through Tauri commands to the Rust backend.

7. **Kill is always reachable.** The kill button is always visible and always functional in the runtime window. Escape key is a hard stop. The user is never trapped in a running loop.

8. **Platform-aware from day one.** Serial port paths differ between macOS (`/dev/cu.*`) and Windows (`COM*`). arduino-cli install paths differ. Process management differs. Every OS-touching code path has explicit macOS and Windows branches.

9. **Design for future embedding.** All communication with Ollama, arduino-cli, and external APIs goes through abstraction layers in the Rust backend. The interfaces are the contract — not the transport.

## Future Roadmap

### CAD Integration
Import `.step`, `.f3d`, or KiCad schematic files. The model reads the BOM or component placement data and auto-populates the manifest. This eliminates manual parts entry for users who design in CAD.

### Multi-board Projects
A single Cuyamaca project coordinates multiple Arduino boards, each with their own manifest, sketch, and serial connection. The runtime model treats them as a unified tool set.

### Vision Pipeline Expansion
Full integration of local vision models (Moondream 2 for lightweight, Llama 3.2 Vision for capable) into the sensor visualization pipeline. The model describes what it sees in camera frames and uses that description as additional context for tool selection.

### Embed Dependencies
Bundle arduino-cli and Ollama directly into the app binary. Both are permissively licensed. Eliminates first-run install entirely.

### Python Serial Scripts
Support uploading Python scripts that control the board via pyserial as an alternative to `.ino` sketches. The code model wraps them into callable tool functions and the Rust backend spawns them as child processes.
